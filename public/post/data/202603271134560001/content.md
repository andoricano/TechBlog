원형은   
var a : Int = 0 이고 메모리 할당부터 앞으로 있을 모든 동작까지 c의 int a = 0;과 똑같습니다. 

후자의 경우 a에 접근해서 타입을 체크하고 그게 값이라면 복사 시 값복사처럼 보이는 동작을 지원하긴 합니다. 그래서 js가 따로 primitive type이라고 명명했죠.
그러나 실제 동작을 보면 a를 호출했을 때 a에 담겨있는 레퍼런스를 확인해서 그 공간에 갑니다. 기능적으로는 레퍼런스 타입이죠?
값복사 자체도 사실 값이 아니라 불변 데이터 저장 공간(primitive type이 저장되는 공간)에서 해당 값을 가리키는 레퍼런스 값을 바꿔주는 방식으로 구현했습니다.
당장 2개의 언어 같은 문법, 같은 개념인데도 내부적으로 너무 다릅니다.

즉 js의 primitive type이라고 불리는 친구들은 사실 primitive type이 아니긴 해 라고 정리를 내리는 것이 이 언어가 사용하는 변수에 대한 이상적인 이해라고 봅니다.
이렇게 말하기 위해선 결국 c부터 타고 올라와야 한다는 거죠.

# 해당 언어들의 범주화
카테고리를 제가 Native, VM, interpreter로 나눴는데 그 이유를 간단하게 설명드리고 가려고 합니다.
위 언어들은 다양하게 범주화될 수 있습니다. 단순하게 OOP기반으로 나눠도 나눠지고, compile vs interpreter로 나눠도 나눠집니다. 

제가 원하는건 전 챕터 말미에 다뤘던 환경까지 다루는겁니다. 언어의 환경을 나누는 가장 큰 범위의 범주가 위 3개라고 봤기 때문에 3가지로 나눴습니다.

## Native Compile
코드를 bin으로 컴파일합니다. 이 bin은 OS에서 직접 처리할 수 있습니다. 그래서 실행할 환경(OS)에 대해 종속적으로 컴파일 해야합니다.
- c/c++ : 보통 직접 컴파일러를 사용하여 bin 파일을 빌드합니다. 이 과정도 사실 발전한 사실이라는 것, 포스트에서 보시죠.
- rust : 언어 자체에 빌드 도구가 내장되어 있어 bin 파일을 c/c++보다 편하게 빌드할 수 있습니다.
- swift : 안 써봐서 잘 모르겠는데 찾아보니까 Apple 다운 컴파일이라고 칭찬이 많네요.
- go : 안 써봐서 잘 모르겠는데 찾아보니까 Google 다운 컴파일이라고 칭찬이 많네요....

## VM
소스 코드를 Byte로 컴파일 합니다. 이 Byte는 OS 상관없이 VM에서 처리하여 같은 소스라면 같은 동작으로 작동하도록 설계되었습니다(라고 배웠으나 맨날 치명적인 부분에서 조금씩 다르게 동작하는 것 같음).
- C# : Roslyn이라는 컴파일러가 소스코드를 IL로 컴파일해줍니다. 빌드된 exe 파일을 .NET Runtime 환경에서 OS 상관없이 실행할 수 있습니다.
- Java : javac 컴파일러가 소스를 bytecode로 컴파일합니다. 얘는 .class 파일을 JVM 환경에 올려 OS 상관없이 실행할 수 있습니다.
- Kotlin : java와 유사합니다.

## Interpreter
- js(ts) : V8이 코드를 실시간으로 해석하며 잘 실행해줍니다.
- py : CPython interpreter가 코드를 한 줄 한 줄 읽으면서 실행해줍니다.

위 특성에 따라 3가지로 범주화했습니다.
정리 순서는 통상적으로 언어에 대해 정리하는 것과 유사하게 일단 변수 먼저 진행하겠습니다.
