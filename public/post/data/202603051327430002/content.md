# Tech Blog의 설계

먼저 Tech Blog를 만든 목적부터 말하자면 나의 포트폴리오를 Notion이나 Naver, git 만으로 표현하기 좀 불편한 부분이 있었기 때문에 WebPage 수준의 커스터마이징이 가능한 환경이 필요했다. 
그래서 위 서비스들을 대체하는 Editor 역할을 수행할 수 있도록 설계했다.

## 설계도
다음은 해당 Tech Blog의 설계도이다. 초기와 현재 그리고 앞으로를 Block Diagram 나타낸다.

### 초기 설계
- 초기 설계 블록 다이어그램
  ![초기 설계 블록다이어그램](/post/data/202603051327430002/images/first_block_diag.png)
정말 간단하다. Post data를 vscode에서 md로 만든 후 HTML+CSS로 보여주기만 했다. 간단하게 할 수 밖에 없었던 이유는 필자가 java script에 대해 무지했기 때문이다. 스스로 제작하면서 다양한 기능에 대한 필요성을 매우 느껴서 js를 적용했고, 이에 따라 js의 작동 원리에 대한 간단한 정리도 할 수 있었다.

### 현재
- 현재 설계 블록 다이어그램
    ![초기 설계 블록다이어그램](/post/data/202603051327430002/images/second_block_diag.png)
    그러나 카테고리의 기능이 필요하다는 조언을 듣고, 해당 기능을 구현하기 위해 js로 구현하려 했으나 유지보수가 굉장히 힘든 프로젝트가 되어가고 있었다. 스스로 어떠한 기능이 있는 웹 페이지를 만들어 본 적이 전무했기에 어떠한 프레임워크나 라이브러리가 필요했다. 이에 따라 가장 인기가 많은 리액트를 사용하기로 했고 그나마 이전에 애플리케이션 개발을 해왔기 때문에 제미나이의 힘을 빌려 현재의 설계를 구현해낼 수 있었다.  
    언젠가 DB-Server-Web 구조로 분리가 되어도 빠르게 수정할 수 Post를 따로 분리했다. Post 틀을 만들거나, category map을 json 으로 만드는 등 유용하게 사용할 script 등도 구비했다.

## 계획 및 방향성

사실 계획이나 방향성이 거의 웹 학습에 가깝다. 필자는 웹 개발을 거의 모르고 웹에 대한 지식 수준도 적다. 심지어 리액트도 거의 처음이다. 해당 웹 페이지는 제미나이를 학대하면서 만들었다. 그래도 해당 블로그를 만들면서 웹 및 리액트에 대한 숙달할 예정이고 이에 따라 컨텐츠까지 더 풍부해질 것은 장담한다.  
필자는 라이브러리나 프레임워크를 접할 때 먼저 사용해본다. 사용하는 과정에서 어떤 의도로 만들었을지 추측하고, 찾아본 후 나의 것으로 만든다. 이러한 과정을 반복하다보면 리액트를 나의 도구로 만들 수 있고 해당 블로그의 컴포넌트와 포스트도 수없이 늘어날 것이다.  
마지막으로 유니티, 언리얼, 안드로이드 등 필자가 거쳐왔던 모든 지식과 프로젝트들을 이곳에 총망라할 예정이다.
