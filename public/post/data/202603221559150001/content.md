# JavaScript Prototype Concept

이전 포스팅에서 JavaScript에 대해 정리할 때 다른 언어와 구별되는 동작과 개념 위주로 정리했다.
타입이라는 개념을 모두 레퍼런스 참조값이라는 구현으로 없애 버렸다는 점에서 참 특이한 케이스다.
도중에 class 대한 타입이 Dictionary라는 것을 확인했고 function, object, array 등과 동작만 다르지 타입은 같다는 것을 property를 추가하고 출력하는 로그로 확인할 수 있었다.
일반적인 언어에서 class를 상속하는 방식을 사용하여 코드 재반복을 막는다.
그렇다면 이 JS에서는 class를 어떻게 상속할까?
사실 class라는 개념이 없었다.
어차피 function이든 object든 간에 id 레퍼런스 가지고 오면 해당 레퍼런스에 접근해서 내부에 모두 접근이 가능했는데 굳이 상속이 필요한가 싶기도 하다.
그러나 이렇게 막 연결하다 보면 그 연결 고리가 진짜 그물망이 되어버려서 감당이 안 될 것이다.
타입에 대한 자유로움을 기조로 하기 때문에 상속은 그 기조에서 엇나가는 행위나 마찬가지다.
그래서 하나의 표준 통로 정했는데 그것이 prototype이다.

## Delegate
Prototype에 대한 사용을 명확하게 이해하기 위해선 Delegate 개념을 조금이나마 숙지해야 한다.
기본적으로 해당 단어는 대리인으로 사용되었고 권력을 부여받은 누군가가 일을 대리한다라는 의미를 가지고 있다. 상속은 실체적인 도구를 받고 해당 도구를 직접 사용하기에 확실히 다르다.  
C#이나 코틀린같이 Delegator라는 문법을 언어 자체에서 지원해주는 언어를 생각해보면 모두 위임을 명시해야만 한다.
A의 a기능을 B에게 위임하면 그때부터 B가 a기능을 상속이나 A의 실체적인 구현이 없어도 사용할 수 있다.
JS의 경우 위임하는 단계를 모두 내부적으로 구현시켜버렸다. 원시타입 변수와 의도적으로 prototype을 끊어버린 객체가 아닌 이상 모두 prototype에 접근할 수 있는 권한이 있는 것이다.
이 말은 즉 prototype이라는 건 어딘가에 단 하나만(권력) 있고 나머지는 모두 참조(대리)만 한다는 것이다.
이제 prototype 개념에 대해 명확하게 확인하고 JS에서 어떻게 구현하고 사용했는지 살펴보자.

# Prototype System
JS를 잘 살펴보면 변수를 통해 다양한 초기화 방법을 제공한다고 느껴질 때가 있는데 prototype이 유독 다양하다.
{}, function, class로 선언하는 것마다 모두 prototype 연결이 달라진다.
prototype으로 객체 간 데이터를 공유하는 것을 prototype chain 이라고 하니 이제 프로토타입을 통해 데이터를 공유하는 것을 프로토타입 체인이라고 쓰겠다.
가장 최상위 Object의 prototype에 어떻게 접근해서 사용하는지 로그로 찍어볼 것이다.

## __proto__와 prototype
로그로 살펴보기 전에 이 2가지는 간단하게 살펴보고 가자.
던더(Doublue Underbar)로 명명된 __proto__와 prototype이 있다.

- \_\_proto\_\_ : 객체가 연결된 프로토타입 체인을 찾는 참조값
- prototype : 프로토타입 체인에 연결해 줄 참조값

참고로 던더를 사용한 이유는 일반 개발자가 괜히 오버라이드하지 말라는 의미.

# JS Inheritance
마지막으로 prototype이라는 데이터 관리 방법이 있는데 왜 상속이라는 개념이 ES6에서 생겼나? 라는 질문이 들 수도 있다.
이유는 단순하다. prototype을 개발자마다 색다르게 사용해서 중구난방이되었기 때문이다.
그래서 개발자들이 표준적으로 데이터를 관리할 때 자주 쓰는 상속을 만들어서 코드 형태로나마 비슷하게 만들어놨다.
