# Unity의 계층 구조

이전에 Component라는 개념이 소프트웨어 개발에 어떻게 녹아들었는지 정리했습니다. 이제 사용법에 대해 정리하려고 합니다. 리액트같은 경우는 컴포넌트를 UI 모델 하나에 대한 관리 단위로 사용하기 때문에 매우 직관적입니다. 그러나 이보다 더 복잡한 모델링 엔진 수준까지 가면 상위 계층이 생기게 되고 컴포넌트의 의미가 미묘하게 다르게 적용됩니다. 제가 유니티를 입문했을 때 가졌던 의문 중 하나가 이 미묘한 차이였기에 유니티의 컴포넌트를 정리하려고 합니다.  

## Unity Component

결론부터 말씀드리면 유니티에서 유니티 컴포넌트는 객체의 동작을 정의하는 단위입니다. 유니티로 만들어진 화면 상에 있는 모든 것들은 컴포넌트로 인해 생겨나고 변화합니다. 유니티의 Scene, GameObject는 뭐하길래 컴포넌트로만 화면이 이루어질까요? 간단하게 계층 구조를 정리하고 갑시다.

### 유니티 계층 구조 요약

| 계층           | 역할              | 설명                                                                            |
| :------------- | :---------------- | :------------------------------------------------------------------------------ |
| **Project**    | 자원 관리         | 게임을 구성하는 모든 에셋(스크립트, 모델, 사운드 등)의 집합소                   |
| **Scene**      | 환경 구성 및 트리 | GameObject Node로 이루어진 Tree를 통해 특정 시점의 게임 환경을 구성             |
| **GameObject** | ID, 관리 단위     | Scene 내에 존재하는 모든 객체의 기본 단위로 Scene을 구성                        |
| **Component**  | 기능 단위         | 실제 데이터와 로직을 정의하여 객체의 동작을 결정하는 부품으로 GameObject를 구성 |
---

## Unity Component와 UI 라이브러리의 Component 차이

리액트나 젯팩 컴포즈같은 경우는 컴포넌트 하나가 바로 환경에 구성 요소로 사용되었죠. 그러나 유니티의 컴포넌트는 GameObject가 없다면 사용할 수 없습니다. 이는 관점의 차이인데 독립적인 객체를 어떤 부품으로 조립하냐 라는 개념에서 시작해야 합니다. 즉 프레임과 환경이 다르다는거죠. 위의 UI 라이브러리들은 프레임이 곧 환경입니다. 그러나 유니티의 component들이 조립될 프레임은 Object입니다. 이제 우리는 유니티와 UI 라이브러리들의 컴포넌트 조립 대상(프레임)이 다른지 파악해야 합니다. 표를 유심히 봤다면 Scene부터 모두 구성이라는 단어로 끝나느 것을 알 수 있습니다. Scene, GameObject, Component 모두 컴포넌트라는 것이죠.  
Scene 자체는 게임 실행 환경을 정의하는 컴포넌트입니다. 구성된 GameObject 노드들을 가지고 자신이 계산해야 할 component들을 찾아 Unity Engine으로 지속적으로 연관된 모든 것을 출력합니다. 이 거대한 조립품들이 모여 하나로 연결된다면 그것이 게임이겠죠?  

Scene의 노드로 구성되는 역할을 맡은 컴포넌트 즉 GameObject들은 Unity Component들을 모아두는 역할을 하는 컴포넌트입니다. 부모 Node 포인터랑 부착된 Component 포인터 리스트를 모아둔 Struct 정도로 보면 됩니다.

Component는 Object를 수정하는 연산을 진행합니다. 예를 들어 Component 내부의 무언가는 독립적으로 실행하기 어렵습니다. 예를 들어 Collision Component의 기능을 Module로 사용하게 된다면 우리는 여러 가지 작업을 수반해야 합니다. Coliision 범위를 코드에서 직접 x,y 좌표로 조정해야 하고 Collision 끼리 부딪히면 어느 정도 튕길지를 또 코드에서 조정해야 합니다. 그러나 Component로 가져오게 되면 interface로 수치만 조정해주면 됩니다. class 내부에 직접 collision을 import 했더라도 메소드 인터페이스로 값만 넣어주면 됩니다.  

## 결론

정리하자면 기본적인 Component Depth가 이미 있기 때문에 해당 Layer 별로 독자적인 Component Name을 달았다는 것이죠. 해당 이름으로 유저 입장에서 그 Layer에서의 Component가 어떤 역할을 하는지 쉽게 유추할 수 있으니 좋습니다.

사담으로 끝내보면 Scene을 추가할 수 있다는 점이 굉장히 매력적이네요. 안드로이드 같은 경우는 Activity가 Scene이라고 보면 되는데 발전을 할 수록 Activity 추가를 지양하는 흐름으로 발전했습니다. 리액트나 플러터같은 경우는 Scene을 추가한다는 개념이 거의 배제되어있습니다. 이러한 개념 차이에서 코드 디자인 아키텍처가 확연하게 차이나는데 이 부분은 추후 포스팅할 수 있으면 좋겠네요.
