# JavaScript의 Object

일반적으로 OOP 개념을 다룬 언어들은 class의 구현을 통한 인스턴스에 접근할 수 있는 pointer value에 reference의 개념을 더해 object를 구현했다.
그러나 JS의 경우 class라는 개념이 없었다. 그러나 JS도 object라는 타입이 있고 키워드도 있다.
같은 개념을 다르게 사용하는 방식을 인지하기 위해서 먼저 JS의 object 초기화 과정을 상세하게 파악할 필요가 있다.

코드 윗 부분은 이전에 간단하게 다뤘던 const object이다. const의 경우 id에 연결되는 값 변경이 잠금이 되는거지, heap에 있는 데이터 영역은 잠금되지 않는다. id object에 접근하면 할당 에러가 발생한다.
여기서 신기한건 Object를 선언했고 해당 변수에 함수를 넣었는데도 정작 constant 에러만 발생한 점을 볼 수 있다. var로 하나 만들고 이것저것 넣어보면 참조값이 다 들어간다.
참조값 들어가는 건 둘째치고 참조하여 바라보고 있는 타입과 다른 행위를 해도 일단 실행이 된다.
여기서 알 수 있는 사실은 동작으로서의 타입만 존재하고, 저장 상태에 대한 타입은 아예 없다는 것이다.  
- ()라는 예약어로 함수 포인터 호출까지 하나 정작 가보니 함수 형태의 데이터가 아니어서 에러를 뱉어낸다.  
- []라는 예약어로 Array형태로 읽어들이려고 하나 정작 가보니 배열 형태의 데이터가 아니어서 에러를 뱉어낸다.  
[]에 접근하는 등의 무언가 동작해서 에러가 잡힌다면 그나마 디버깅이 가능하다.

다만 여기서 문제는 선언도 하지 않은 Property에 접근하는 케이스이다.
심지어 class로 만들었을 경우 [] 호출이 가능하다.
class라는 예약어를 사용해 instance를 만들었을 때 []로 접근하거나 선언하지 않은 property를 호출하면 에러가 아닌 undefined가 출력된다.

이 부분에 대한 확실한 답을 내리기 위해서 내부 구현이 어떻게 되었는지 파악해야 한다.
일단 공식 문서 상에 당당히 Dictionary라고 표현된다.

>JavaScript objects mostly behave like dictionaries, with string keys and arbitrary objects as values.

지금까지 위의 변수를 만드는 행위는 id라는 키를 가지고 객체를 참조할 수 있는 참조값을 만들어서 Dictionary로 관리하여 인스턴스에 연결하는 행위였고, 찾을 때도 Dictionary로 찾았다.
Java나 C#의 경우 선언한 프로퍼티나 메소드를 못 찾을 경우 빌드조차 되지 않는다.  
JS와의 차이는 뭘까?
누구나 알고 있듯이 JS는 런타임에서 코드를 실행하는 인터프리터 언어이기 때문이다.
컴파일러 언어는 메모리에 정확한 사이즈로 할당하고 그 메모리 주소를 관리한다.
예를 들어 x0000~x000F까지 값이 있다면 그 안에 0004, 000A, 000F 선언한 대로 쪼개진다.
그러나 JS의 경우 메모리에 자기 프로퍼티 개수 만큼 참조값 단위로 주소를 만든다.
위 user라는 Object에 name 주소가 있고, age 주소가 있다.
정확히 "Kim"을 가리키는 주소와 30을 가리키는 주소가 있는 것이고, 클래스 자체 메모리가 7byte가 아니라 해쉬값 2개인 16byte다.
