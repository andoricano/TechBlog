# Component의 사전적 의미와 소프트웨어 개발에서의 발전

컴포넌트처럼 널리 사용되는 용어들은 의미가 비슷해도 약간씩 사용면에서 차이를 보이기 때문에 그 뿌리를 명확히 알아야 합니다. 그래서 가장 근본적인 의미인 사전적인 단어를 먼저 정리합니다.

- "a part or element of a larger whole, especially a part of a machine or vehicle."  
- "(구성) 요소, 부품"  
- "component part = component가 될 수도 있다."

사전적 의미는 기계와 매우 밀접합니다. 다만 기계 자체에서 따온 것은 아니라고 합니다. 1968년 NATO 소프트웨어 공학 회의에서 더글러스 매킬로이 박사가 해당 개념을 발표했다고 합니다. 핵심 주장으로 "소프트웨어도 하드웨어처럼 미리 검증된 표준화된 부품(Components)를 만들어 이를 조립하는 방식으로 제공해야 한다."가 있습니다. 부품을 어떻게 정의하느냐에 따라 다르겠으나 확실한 건 결국 
"무언가에서 사용(조립)할 수 있는 코드" 부품은 컴포넌트라고 보면 되겠습니다.
물론 절차 지향 언어나 플랫폼에서 모듈, 라이브러리, 패키지 등 위 개념을 사용하고 있었으나 명확한 시스템 아키텍처 패러다임으로 정립되지 않았습니다.  

컴포넌트를 조립해 소프트웨어를 구축하는 방식은 1990년대 객체지향 프로그래밍이 개발 시장에서 주류 패러다임으로 자리 잡으면서 다양한 플랫폼에서 컴포넌트 모델의 형태로 제공되기 시작했습니다.
OOP는 시스템을 객체 단위로 구조화하고 객체는 기능에 따라 설계됩니다. 이러한 객체를 모아 독립적인 컴포넌트를 구현하는 방식까지 자연스럽게 발전하게 되었습니다.  

결국 2010년대에 이르러 컴포넌트라는 개념이 설계나 개념 단계에서 머무르지 않고 플랫폼의 기반 구조로 사용되기 시작합니다. C#을 사용하는 Unity, JS를 사용하는 React, Compose/kotlin을 사용하는 Android 등 UI 개발에서 폭발적으로 확산되죠. 해당 Component를 다루는 방식은 약간씩 다릅니다.
Unity에선 Object에 Component를 붙여 기능을 추가합니다. React에선 함수 형태를 UI로 정의하며 이 함수 단위를 Component로 관리합니다. Jetpack  Compose도 이와 유사한 형태로 관리하죠.
구현 방식은 서로 다르지만, 기능을 독립적인 단위로 분리하고 이를 조합하여 시스템을 구성한다는 컴포넌트의 핵심 개념은 공통적으로 유지되고 있습니다.

마지막으로 코드를 기능적으로 분리하는건데 왜 Module과 개념이 다른지 설명하고 마무리짓겠습니다. 사실 제가 이해가 안 되어서요.. Module은 말 그대로 코드를 관리하기 위한 단위입니다. 개발자를 위한 개념이라는 겁니다. 복습을 위해 잠깐 정리합시다.

## * 코드 관리 관점 (Development & Maintenance)
개발자가 코드를 작성하고 관리하기 위한 물리적인 파일 및 디렉토리 구조입니다.

* **Package**: 모듈들을 논리적으로 계층화하여 묶어놓은 폴더/디렉토리 단위.
* **Module**: 유사한 기능을 기준으로 이루어진 파일 단위 코드 묶음.
* **Class**: 데이터와 기능을 캡슐화하여 객체를 생성하는 틀.
* **Function**: 특정 입력을 받아 로직을 수행하고 결과를 반환하는 최소 실행 단위.



---

## * 시스템 구성 관점 (Architecture & Composition)
시스템이 어떻게 조립되고 배포되는지, 즉 기능적 부품으로서의 구조입니다.

| 구분          | 개념 정의                                               | 역할                           |
| :------------ | :------------------------------------------------------ | :----------------------------- |
| **System**    | 목적을 달성하기 위한 전체 구성 요소의 집합체            | 서비스의 최종 완성 형태        |
| **Framework** | 컴포넌트나 모듈을 배치하고 동작하게 하는 뼈대 혹은 규칙 | 시스템의 기반 설계도           |
| **Component** | 독립적 배포/교체 단위 (인터페이스 소통)                 | 시스템의 큰 부품 (독립성 보장) |
| **Module**    | Component를 구성하는 내부 기능 단위 (인증, 통신 등)     | 컴포넌트의 논리적 세분화       |
| **Function**  | 특정 입력을 받아 로직을 수행하는 최소 실행 단위         | 코드 수준의 실질적 연산자      |

### 핵심 차이 요약
* **코드 관리 관점**: "코드를 어떻게 파일로 쪼개어 효율적으로 유지보수할 것인가?"에 집중합니다.
* **시스템 관점**: "기능적으로 분리된 덩어리들을 어떻게 조합하여 유연한 시스템을 만들 것인가?"에 집중합니다.

정리를 해도 약간 헷갈리네요. 추후에 React의 Component에 대한 개념과 사용을 포스팅할 건데 그때 예시를 스스로 만들어가면서 개념을 확립해야겠습니다.
